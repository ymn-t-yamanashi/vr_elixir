# スプリント1実施ルール確定版

## 1. Docker実行方式
- ホスト実行は禁止し、Docker内で実行する。
- 通常起動は `docker compose up app` を使い、コンテナ内で `mix run --no-halt` を実行する。
- テスト/品質系は `docker compose run --rm app ...` で実行する。
- `make` 化する方針とする（実装は壁打ち後に実施）。

## 2. 接続確立条件
- WebSocket が `open` になった後、`requestSessionData` の成功応答を受信した時点で「接続確立」とする。
- 上記のどちらかが失敗した場合は未接続扱いとする。

## 3. 再接続仕様
- 再接続中の `call/2` はキューイングしない。
- 再接続中の `call/2` は即時 `{:error, :not_connected}` を返す。

## 4. リクエスト対応付け
- `messageId` は `UUID v4` を使用する。
- 送信時に `pending` へ登録し、同一 `messageId` の応答で解決する。
- タイムアウト時は `{:error, :request_timeout}` を返し、`pending` から削除する。
- 未知の `messageId` 応答は `warn` ログのみ出力し、処理しない。

## 5. ログ仕様
- 共通項目: `timestamp`, `level`, `event`, `message_id`, `$type`
- 接続系項目: `host`, `port`, `state`, `reason`
- API系項目: `success|failure`, `error_reason`, `elapsed_ms`
- マスク対象: 認証情報、個人情報、巨大ペイロード本文
- レベル目安:
  - `info`: 接続/切断、送信開始、成功
  - `warn`: 再接続、タイムアウト、未知 `messageId`
  - `error`: デコード失敗、transport失敗、内部例外

## 6. API最小セット
- 対象コマンド:
  - `requestSessionData`
  - `addSlot`
  - `updateSlot`
  - `addComponent`
  - `updateComponent`
  - `removeComponent`
  - `removeSlot`
- 統一IF: `call(client, type, payload)`
- 戻り値: `{:ok, response} | {:error, reason}`

## 7. E2E固定手順
1. `requestSessionData` 成功
2. `addSlot`
3. `updateSlot`
4. `addComponent`
5. `updateComponent`
6. `removeComponent`
7. `removeSlot`
8. 途中で切断を1回発生させ、再接続後に再実行成功

## 8. 失敗系期待値
- 不正 `$type`: `{:error, :invalid_request}` または `{:error, {:remote_error, code, message}}`
- 存在しない `slotId` / `componentId`: `{:error, :not_found}` または `{:error, {:remote_error, code, message}}`
- 応答なしタイムアウト: `{:error, :request_timeout}`
- 接続断中の `call/2`: `{:error, :not_connected}`

## 9. 追加確定事項
- `payload` 必須キー:
  - `requestSessionData`: 必須なし（`{}`）
  - `addSlot`: `parent_id`, `name`
  - `updateSlot`: `slot_id`（更新項目は1つ以上必須）
  - `addComponent`: `slot_id`, `component_type`
  - `updateComponent`: `component_id`, `members`
  - `removeComponent`: `component_id`
  - `removeSlot`: `slot_id`
- `request_timeout_ms` は `10000` 固定とし、`call` 単位の上書きは行わない。
- 切断検知トリガーは `close frame` と `tcp error` のみ採用する（`pong timeout` は採用しない）。
- `type` は `$type` 文字列を直接使用し、許可リストで厳密検証する。非許可値は `{:error, :invalid_request}` とする。
- エラーログは通常テキストの1行形式で出力する。
- `payload` の内部キー命名は `snake_case` で統一する（例: `slot_id`, `component_id`）。
- バリデーション責務は、`Scene.call/3` で入口チェック、`Protocol` で送信前の詳細チェックを行う。
