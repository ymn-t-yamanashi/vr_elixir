# 詳細設計

## 1. 接続方式
- 本仕様の接続方式は `ResoniteLink` のみを採用する。
- ResoniteLink は公式の `WebSocket + JSON` プロトコルとして利用する。

## 2. 通信方式
- `ResoniteLink` 経由でコマンド/応答を送受信する。
- メッセージ形式はJSONを採用する。
- メッセージは ResoniteLink 公式モデル（`$type` ベース）に準拠する。

## 3. Elixir実装詳細

### 3.1 モジュール分割
- `ResoniteLinkEx.Client`
  - ResoniteLink接続管理
  - 再接続（指数バックオフ）
  - 送受信ループ
- `ResoniteLinkEx.Protocol`
  - コマンドJSONエンコード
  - 応答JSONデコード
  - バリデーション
- `ResoniteLinkEx.Scene`
  - 基本API提供（ResoniteLinkで利用可能な全命令）

### 3.2 想定公開API

- `ResoniteLinkEx.Scene.call/2`（`$type` を含む任意のResoniteLink命令を送信）
- `ResoniteLinkEx.Scene.call!/2`
- `ResoniteLinkEx.Scene.supported_commands/0`（対応命令一覧の取得）

### 3.3 ライブラリ

## 4. メッセージ仕様（公式モデル準拠）
- フォーマット: 1リクエスト1行JSON
- 共通フィールド:
  - `$type`: メッセージ種別
  - `id`: データモデル要素識別子（Slot/Component）
- 本PJの基本APIは、ResoniteLinkで利用可能な `$type` 命令すべてを対象とする。

### 4.0 全命令一覧（ResoniteLink公式定義）
以下は公式 `Message` 派生として定義される `$type` 命令の一覧。

#### セッション/バッチ
| `$type` | 詳細 |
|---|---|
| `requestSessionData` | 接続先セッションの状態情報（セッションID、ルート参照など）を取得し、以後の操作対象解決に使う |
| `dataModelOperationBatch` | 複数のデータモデル操作を1メッセージで送信し、順序を保ってまとめて適用する |

#### Slot操作
| `$type` | 詳細 |
|---|---|
| `getSlot` | 指定した `slotId` のSlotデータ（名前、Transform、親子関係など）を読み出す |
| `addSlot` | 親Slot配下に新しいSlotを作成し、初期値（名前・位置など）を設定する |
| `updateSlot` | 既存Slotのメンバーを部分更新し、位置・回転・スケールや名前などを変更する |
| `removeSlot` | 指定したSlotを階層ごと削除し、配下の要素も含めてワールドから取り除く |

#### Component操作
| `$type` | 詳細 |
|---|---|
| `getComponent` | 指定した `componentId` のComponent構成と現在値を取得する |
| `addComponent` | 対象Slotに指定型のComponentを追加し、必要な初期メンバーを与える |
| `updateComponent` | Componentの `members` を部分更新し、フィールド値や参照値を変更する |
| `removeComponent` | 指定ComponentをSlotから除去し、機能・状態の付与を解除する |

#### Texture2D
| `$type` | 詳細 |
|---|---|
| `importTexture2DFile` | 画像ファイルを読み込み、2Dテクスチャアセットとして生成して参照可能にする |
| `importTexture2DRawData` | ピクセル生データとフォーマット情報から2Dテクスチャを生成する |
| `importTexture2DRawDataHDR` | HDRピクセル生データ（高ダイナミックレンジ）から2Dテクスチャを生成する |

#### Cubemap
| `$type` | 詳細 |
|---|---|
| `importCubemapFiles` | 面ごとに用意した複数画像ファイルからキューブマップアセットを作成する |
| `importCubemapFileWithRegions` | 1枚画像内の領域定義を使って各面を切り出し、キューブマップを作成する |
| `importCubemapRawData` | 各面の生データを直接与えてキューブマップを生成する |
| `importCubemapRawDataHDR` | HDR生データを用いて高精度キューブマップを生成する |

#### Mesh
| `$type` | 詳細 |
|---|---|
| `importMeshJSON` | 頂点・法線・UV・サブメッシュ情報をJSONで受け取りメッシュアセット化する |
| `importMeshRawData` | バイナリ生データ形式の頂点/インデックス情報からメッシュを生成する |

#### Audio
| `$type` | 詳細 |
|---|---|
| `importAudioClipFile` | 音声ファイルを読み込み、再生可能なオーディオクリップとして登録する |
| `importAudioClipRawData` | PCM等の生データとサンプリング条件からオーディオクリップを生成する |

#### Reflection
| `$type` | 詳細 |
|---|---|
| `getTypeDefinition` | 指定型の構造情報（メンバー、継承、属性など）を取得する |
| `getGenericTypeDefinition` | ジェネリック型の型引数条件や展開情報を取得する |
| `getEnumDefinition` | 列挙値の名前・数値・フラグ情報を取得する |
| `getComponentDefinition` | Component型のメンバー定義と更新可能項目を取得する |
| `getSyncObjectDefinition` | SyncObjectの同期対象メンバーや構造定義を取得する |
| `getComponentTypeList` | 現在利用可能なComponent型一覧を取得し、追加候補選定に使う |

#### Sync Method
| `$type` | 詳細 |
|---|---|
| `callSyncMethod` | 対象インスタンスの同期メソッドを引数付きで実行し、戻り値を受け取る |
| `callStaticSyncMethod` | 型に紐づく静的同期メソッドを実行し、グローバル処理を呼び出す |

### 4.1 代表例: Slot生成（addSlot）

制約:
- `data.name.value` は必須項目。
- `name` はワールド内で一意であること（重複禁止）。
- 同名が存在する場合、生成は失敗（`conflict`）として扱う。

```json
{
  "$type":"addSlot",
  "data":{
    "id":"MySDK_0",
    "parent":{"$type":"reference","targetId":"Root"},
    "name":{"$type":"string","value":"BoxA"},
    "position":{"$type":"float3","value":{"x":0,"y":1,"z":0}}
  }
}
```

### 4.2 代表例: Transform更新（updateSlot）

```json
{
  "$type":"updateSlot",
  "data":{
    "id":"MySDK_0",
    "position":{"$type":"float3","value":{"x":1,"y":1,"z":0}},
    "rotation":{"$type":"floatQ","value":{"x":0,"y":0,"z":0,"w":1}},
    "scale":{"$type":"float3","value":{"x":1,"y":1,"z":1}}
  }
}
```

### 4.3 代表例: コンポーネント値更新（updateComponent）

```json
{
  "$type":"updateComponent",
  "data":{
    "id":"MySDK_1",
    "members":{
      "Scalable":{"$type":"bool","value":false}
    }
  }
}
```

### 4.4 代表例: Slot削除（removeSlot）

```json
{"$type":"removeSlot","slotId":"MySDK_0"}
```

## 5. 応答仕様

### 5.1 方針
- 応答形式は ResoniteLink 公式モデルに従う。
- 独自の `ok` / `error` 形式は採用しない。

### 5.2 エラー分類（推奨）
- `invalid_request`: 必須項目不足/型不一致
- `not_found`: target/componentが存在しない
- `conflict`: 同名生成など競合（例: 重複 `name`）
- `internal_error`: 実行時例外

## 6. Resonite側実装要件
- ResoniteLink をワールドで有効化する。
- ResoniteLinkPermissions で read/write 権限を設定する。
- 受信JSONは ResoniteLink 公式モデルとして扱う（独自 `op` 形式は使わない）。

## 7. 接続・信頼性
- 接続断時は自動再接続。
- 送信中断時は失敗を呼び出し元へ通知。
- タイムアウトを設け、未応答リクエストを解放。
- ResoniteLink は beta であり、破壊的変更の可能性を前提にバージョン追従する。
- 高頻度リアルタイム制御は用途外とし、必要時は in-world 機構の利用を検討する。

## 8. 検証詳細
- 主要命令を連続100回実行して欠落なし。
- 通信断後に自動復帰し再送可能。
- 不正JSON/不正`$type`で失敗応答を返す。
- 同名オブジェクト時の挙動（拒否）を仕様通り確認。

## 9. 今後の拡張
- メッセージに `version` を導入。
- 識別子を `name` からGUID中心へ移行。
- 認証トークンや許可操作制限を追加。
