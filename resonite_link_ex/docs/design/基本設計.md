# 基本設計

## 1. 目的
ElixirアプリからResoniteワールド内オブジェクトを外部操作できるようにする。

## 1.1 プロジェクト識別子
- Mixプロジェクト名は `:resonite_link_ex` とする。
- ルートモジュール名は `ResoniteLinkEx` とする。

## 1.2 実行環境基準
- Elixir は `1.19.4` を使用する。
- Erlang/OTP は `28.5` を使用する。
- 接続先ホストは `localhost` を標準とする（`127.0.0.1` は使用しない）。

## 2. 対象範囲
本システムで提供する操作は、ResoniteLinkで利用可能な命令すべてを対象とする。
これらの命令群を本PJでは「基本API」と定義する。

## 3. できること
- ElixirからResonite内にオブジェクトを新規作成できる。
- 新規作成時に以下の基本オブジェクトを指定できる。
  - `cube`（立方体）
  - `sphere`（球）
  - `cylinder`（円柱）
  - `capsule`（カプセル）
  - `ring`（リング）
  - `grid`（グリッド）
  - `plane`（平面）
  - `quad`（2D板ポリゴン）
  - `text`（テキスト表示）
  - `empty`（空オブジェクト）
- 既存オブジェクトの位置・回転・スケールを外部から変更できる。
- 対象コンポーネントの値を外部から更新できる。
- 不要なオブジェクトを外部から削除できる。
- 操作結果を応答として受け取り、成功/失敗を判定できる。
- 接続断が発生しても復旧後に操作を再開できる。
- 操作命名を ResoniteLink 公式（`$type`）に統一して運用できる。
- ResoniteLinkで公開されている命令を追加仕様なしで順次利用できる。

## 4. システム構成
- Elixirアプリ
  - 外部制御APIを提供し、Resoniteへコマンド送信する。
- Resonite側受信層
  - 外部入力を受信し、ワールド操作に変換する。
- 通信路
  - ResoniteLink のみを使用して、ElixirとResonite間のコマンド/応答を伝送する。

## 5. 設計方針
- 送受信はリクエスト/レスポンス型のプロトコルで統一する。
- 操作コマンドは ResoniteLink 公式命名（`$type`）を使用する。
- ResoniteLinkで利用可能な命令はすべて基本APIとして実装対象に含める。
- 接続方式は ResoniteLink のみとし、他方式は採用しない。
- オブジェクト生成時は `name` 指定を必須とし、同名の重複生成を禁止する。
- 接続断や不正コマンドを考慮したエラーハンドリングを行う。
- 段階的導入（移動操作 → 生成/削除 → 値更新）でリスクを下げる。

## 6. 非機能要件
- 接続断時に自動復旧できること。
- 連続更新時に取りこぼしを最小化すること。
- 失敗時に原因を判別可能な応答を返すこと。

## 7. 運用・拡張方針
- 初期は操作対象を限定し、安定後に対象オブジェクトや操作種別を拡張する。
- オブジェクト識別、認証、運用制約は環境に応じてポリシー化する。
- 詳細技術仕様は `詳細設計.md` に定義する。

## 8. マイルストーン
1. ResoniteLinkで利用可能な命令一覧を最新化し、基本API一覧として確定。
2. 命令ごとの入出力モデルをElixir側に実装。
3. 命令ごとのエラー処理と再接続時の継続性を実装。
4. 主要命令群の結合試験を実施。
5. 命令追加時の追従手順を運用化。
