# スプリント3 サンプルプログラム案

## 目的
- ここにスプリント3用サンプルプログラムの案を整理する。
- スプリント3で実装した以下を、1本のサンプルで理解できるようにする。
  - `move_slot_by_name/4`（`name` 指定の座標移動）
  - `delete_slot_by_name/3`（`name` 指定の削除）
  - `NameResolver.resolve_slot_id/3` の考え方（`name -> slot_id` 解決）
  - `getSlot` を既存関数と同粒度で扱う方針
  - 互換API（`move_slot/3` / `delete_slot/2`）の位置づけ

## サンプル方針（理解優先）
- 方針1: 「生成 -> nameで座標移動 -> nameで削除」の最短導線を見せる。
- 方針2: 1ファイルで完結し、ログで処理ステップを追えるようにする。
- 方針3: 実装詳細を隠しすぎず、`name` 解決が入るポイントをコメントで明示する。
- 方針4: 各行動の直後に `Process.sleep(1_000)` を入れ、画面変化とログを追いやすくする。

## 最小シナリオ案
1. クライアント起動と接続準備。
2. 図形生成（例: cube、`name: "Sprint3Cube"`）。
   - `Process.sleep(1_000)`
3. `move_slot_by_name/4` で座標移動（`position` を変更）。
   - `Process.sleep(1_000)`
4. `delete_slot_by_name/3` で削除。
   - `Process.sleep(1_000)`
5. 各ステップの結果を `IO.inspect/2` で表示。

## 学習ポイント（このサンプルで伝えること）
- 呼び出し側は `slot_id` を直接知らなくても `name` 指定で操作できる。
- 内部では `name -> slot_id` 解決を通して `updateSlot` / `removeSlot` を呼ぶ。
- `getSlot` は特別扱いせず、既存コマンドと同じレベルで利用する。
- 旧APIは互換維持のため残るが、サンプルでは新APIを主に使う。

- 生成する図形: `cube` 固定
- `parent_name`: 指定なし（既定動作を確認する）
