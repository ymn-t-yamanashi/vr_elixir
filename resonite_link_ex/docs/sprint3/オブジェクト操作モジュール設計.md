# オブジェクト操作モジュール設計（スプリント3）

## 1. 目的
- 生成済みオブジェクトの移動・削除を一貫したI/Fで実行できるようにする。
- 利用者は基本的に `name` 指定で操作し、内部で `slot_id` 解決して実行する。
- オブジェクト生成時の既定親を `ResoniteLinkEx` に統一し、未存在時は自動生成する。
- 既存互換性維持のため、`parent_name` 明示指定時は指定値を優先する。

## 2. 対象範囲
- 対象処理:
  - Slot移動（`updateSlot`）
  - Slot削除（`removeSlot`）
  - 既定親スロット解決（`ResoniteLinkEx`）
  - 既定親スロットの自動生成（必要時）
- 対象外:
  - 回転・スケール変更
  - 一括操作・アニメーション移動
  - Undo/履歴管理

## 2.1 Resoniteの構造前提（概念）
- Resoniteのシーン階層は、基本的に `Slot` の親子ツリーで構成される。
- 見た目や挙動などの機能は、`Slot` に付与される `Component` として表現される。
- 実際の値・状態は `Component` / `SyncObject` のメンバーとして保持される。
- 本スプリントでは、この前提に基づき「公開APIは `name` / `parent_name` で操作し、内部で対象 `Slot` を解決して命令送信する」方針を採用する。

## 3. モジュール構成
- 既存拡張: `ResoniteLinkEx.Shapes`
  - 既定親スロット解決・自動生成フローを追加する。
- 追加モジュール案: `ResoniteLinkEx.Objects`
  - 移動・削除APIを公開し、`Scene.call/3` の呼び出しを集約する。
  - 旧API互換レイヤー（`slot_id` / `parent_id` 受け取り）を同居させ、新APIへ委譲する。
- 依存モジュール:
  - `ResoniteLinkEx.Scene`（命令呼び出し）
  - `ResoniteLinkEx.Protocol`（payload妥当性）
  - `ResoniteLinkEx.Transport`（送信）

## 4. 公開I/F案

### 4.1 オブジェクト移動API
- `move_slot_by_name(client_or_transport, name, position, opts \\ [])`
  - `name`: `String.t()`（必須）
  - `position`: `%{"x" => number(), "y" => number(), "z" => number()}`（必須）
  - `opts`:
    - `:parent_name`（同名解決の探索範囲を絞る任意条件）
  - 送信命令: `updateSlot`
  - 内部処理: `name -> slot_id` を解決して送信
  - payload: `%{slot_id: resolved_slot_id, position: position}`
  - 戻り値:
    - 成功: `{:ok, %{type: "updateSlot", payload: map()}}`
    - 失敗: `{:error, :invalid_request | :not_found | :ambiguous_name | term()}`

### 4.2 オブジェクト削除API
- `delete_slot_by_name(client_or_transport, name, opts \\ [])`
  - `name`: `String.t()`（必須）
  - `opts`:
    - `:parent_name`（同名解決の探索範囲を絞る任意条件）
  - 送信命令: `removeSlot`
  - 内部処理: `name -> slot_id` を解決して送信
  - payload: `%{slot_id: resolved_slot_id}`
  - 戻り値:
    - 成功: `{:ok, %{type: "removeSlot", payload: map()}}`
    - 失敗: `{:error, :invalid_request | :not_found | :ambiguous_name | term()}`

### 4.3 図形生成の既定親解決（既存拡張）
- `spawn_shape/3` の `opts`:
  - `:parent_name` 明示時: 指定値をそのまま利用する。
  - `:parent_name` 未指定時: `"ResoniteLinkEx"` を既定親として利用する。
  - `"ResoniteLinkEx"` スロット未存在時: 作成命令を先行送信してから配下生成する。

### 4.4 旧API互換（段階移行）
- 旧公開I/F（当面維持）:
  - `move_slot(client_or_transport, slot_id, position)`
  - `delete_slot(client_or_transport, slot_id)`
  - `spawn_shape(..., parent_id: "...")`
- 方針:
  - 旧I/Fは内部で新I/Fまたは新解決経路に変換して実行する。
  - 呼び出し時に非推奨警告（deprecation）を出す。
  - ドキュメント上は新I/F（`name` / `parent_name`）を正とする。

## 5. 内部仕様

### 5.1 移動シーケンス
1. 入力検証（`name` 非空文字列、`position` が `x/y/z` 数値）
2. `name -> slot_id` 解決（必要に応じて `parent_name` 条件で絞り込み）
3. `Scene.call(client, "updateSlot", %{slot_id: resolved_slot_id, position: position})`
3. `{:ok, result}` / `{:error, reason}` をそのまま返す

### 5.2 削除シーケンス
1. 入力検証（`name` 非空文字列）
2. `name -> slot_id` 解決（必要に応じて `parent_name` 条件で絞り込み）
3. `Scene.call(client, "removeSlot", %{slot_id: resolved_slot_id})`
3. `{:ok, result}` / `{:error, reason}` をそのまま返す

### 5.4 name解決ポリシー
1. `name` 完全一致で候補を取得する。
2. 候補0件は `{:error, :not_found}` を返す。
3. 候補2件以上は `parent_name` 指定がなければ `{:error, :ambiguous_name}` を返す。
4. `parent_name` 指定で1件に絞れた場合のみ `slot_id` を採用する。

### 5.3 既定親スロット解決シーケンス
1. `parent_name` が指定されていればその値を採用する。
2. 未指定なら `ResoniteLinkEx` を既定親IDとして解決する。
3. 既定親が未存在なら `addSlot` で作成する（冪等化: 既に存在する場合は成功扱い）。
4. 図形生成本処理へ進む。

## 6. エラー方針
- 入力値不正:
  - `{:error, :invalid_request}`
- name解決失敗:
  - `{:error, :not_found}`
  - `{:error, :ambiguous_name}`
- 通信失敗・未接続:
  - `{:error, :not_connected}` / `{:error, :transport_error}`
- Resonite側失敗:
  - `{:error, {:remote_error, detail}}`
- 既定親自動生成失敗:
  - `{:error, :parent_slot_prepare_failed}`（内部理由をログに残す）

## 6.1 互換性ポリシー
- スプリント3では旧APIを削除しない（互換優先）。
- 旧API利用時は警告を出し、新APIへ移行を促す。
- 削除タイミングは次スプリント計画で明示し、最低1スプリントの移行期間を確保する。

## 7. テスト方針
- 単体テスト:
  - `move_slot_by_name/4` 正常系・異常系
  - `delete_slot_by_name/3` 正常系・異常系
  - 旧API（`move_slot/3` `delete_slot/2`）の互換動作と警告出力
  - `name -> slot_id` 解決の正常系
  - `not_found` / `ambiguous_name` の異常系
  - `parent_name` 未指定時に `ResoniteLinkEx` を使うこと
  - `parent_name` 明示時に指定値が優先されること（互換性回帰防止）
  - `ResoniteLinkEx` 未存在時の自動生成フロー
- 実機確認:
  - 生成 -> 移動 -> 削除の一連実行
  - `parent_name` 指定あり/なし双方で配置先を確認

## 8. スプリント3完了判定との対応
- 本設計は `スプリント3実施計画` の以下DoDに対応する:
  - Name指定の移動API (`updateSlot`) と削除API (`removeSlot`) の実装
  - `parent_name` 既定値を `ResoniteLinkEx` へ変更
  - `ResoniteLinkEx` 未存在時の自動生成
  - `parent_name` 明示指定時の互換性維持
