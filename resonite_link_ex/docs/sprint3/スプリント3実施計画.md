# スプリント3実施計画

## 1. 目的
- オブジェクトの移動、削除を実装する
- ResoniteLinkExで作成したオブジェクトはResoniteLinkEx配下に作成する
  - parent_nameが無い場合はResoniteLinkExをparent_nameとする
- ResoniteLinkExが無い場合は自動生成する
- 既存の互換性を維持するためにparent_name指定は維持する

## 2. 対象範囲
### In
- 生成済みオブジェクトを `name` 指定で移動するAPIを追加し、内部で `slot_id` を解決して `updateSlot` を送信できるようにする。
- 生成済みオブジェクトを `name` 指定で削除するAPIを追加し、内部で `slot_id` を解決して `removeSlot` を送信できるようにする。
- `name -> id` 解決を共通化する `NameResolver`（仮称）を追加し、移動・削除APIから共通利用する。
- 既存互換性維持のため、旧API（`slot_id` / `parent_id` 前提）は当面併存させ、内部で新API経路へ変換する。
- オブジェクト生成時、`parent_name` 未指定の既定値を `Root` から `ResoniteLinkEx` へ変更する。
- `ResoniteLinkEx` スロットが存在しない場合は自動生成する。
- 既存互換性維持のため、`parent_name` 明示指定時は指定値を優先し、従来挙動を維持する。
- 必須入力（`name`、座標）と異常系（不正型・必須不足・同名曖昧）のバリデーションを実装する。
- 移動→削除の一連操作を確認できるサンプル実行コードを追加する。
- 単体テストと品質ゲートを通し、再現可能な実行手順をドキュメント化する。

### Out
- 回転・スケール変更など、移動以外の変形操作。
- 複数オブジェクト一括操作、アニメーション移動、高度な同期制御。
- 新規図形種別の追加。

## 3. 完了条件（DoD）
- `move_by_name` API が `name` と `position`（x/y/z）を受け取り、内部解決した `slot_id` で `updateSlot` の期待payloadを送信できる。
- `delete_by_name` API が `name` を受け取り、内部解決した `slot_id` で `removeSlot` の期待payloadを送信できる。
- `parent_name` 未指定時に `ResoniteLinkEx` 配下へ生成され、必要時は `ResoniteLinkEx` スロットが自動生成される。
- `parent_name` 明示指定時は指定値が優先され、既存の `parent_name` 指定フローが回帰しない。
- 旧API呼び出し（`slot_id` / `parent_id`）が継続利用でき、非推奨警告を出しつつ同等動作する。
- 不正入力時に `{:error, :invalid_request}` を返すテストが追加され、成功/失敗ケースを網羅する。
- `mix format --check-formatted` / `mix compile --warnings-as-errors` / `mix check.docs` / `mix credo --strict` / `mix test --cover` がすべて成功する。
- 移動→削除の実行手順が `docs/sprint3` 配下に記載され、同手順で再実行できる。

## 4. 実装タスク（最小単位）
1. Name指定移動API設計と実装
- 成果物: `ResoniteLinkEx` 公開API、および `name -> slot_id` 解決を含む移動処理モジュール。
- 確認方法: `name` 指定で `updateSlot` が送信され、payload に解決済み `slot_id` と `position` が反映されることを単体テストで確認する。

2. Name指定削除API設計と実装
- 成果物: `ResoniteLinkEx` 公開API、および `name -> slot_id` 解決を含む削除処理モジュール（`removeSlot` 呼び出し）。
- 確認方法: `name` 指定で `removeSlot` が送信され、payload に解決済み `slot_id` が反映されることを単体テストで確認する。

3. 共通NameResolver設計と実装
- 成果物: `name -> id` 解決の共通モジュール（例: `ResoniteLinkEx.NameResolver`）と、`not_found` / `ambiguous_name` 判定ロジック。
- 確認方法: 一意名解決、未検出、同名曖昧、`parent_name` 絞り込みが単体テストで検証されることを確認する。

4. 既定親スロットの設計と実装（`ResoniteLinkEx`）
- 成果物: `parent_name` 未指定時に `ResoniteLinkEx` を既定親として扱う処理、および未存在時の自動生成処理。
- 確認方法: 未指定時の生成先が `ResoniteLinkEx` になること、未存在時に自動生成後に配下生成されることをテストで確認する。

5. 互換性維持テスト追加（`parent_name` 明示指定）
- 成果物: `parent_name` 指定時の優先挙動を担保する回帰テスト。
- 確認方法: 明示指定値がそのまま送信payloadへ反映されることをテストで確認する。

6. 旧API互換レイヤー追加（段階移行）
- 成果物: `slot_id` / `parent_id` 入力を受け取って新API経路に委譲する互換レイヤー、非推奨警告ログ。
- 確認方法: 旧API呼び出しで従来どおり成功し、警告が出ることをテストで確認する。

7. バリデーションと異常系テスト追加
- 成果物: 必須不足・型不正・同名曖昧時のバリデーション、対応テストケース。
- 確認方法: `mix test` で `invalid_request` / `not_found` / `ambiguous_name` 系テストが成功することを確認する。

8. サンプル実行コード追加
- 成果物: 生成したオブジェクトを移動後に削除するサンプル（`examples/` 配下）。
- 確認方法: `mix run` 実行で、移動と削除命令が順に送信されるログを確認する。

9. 実行手順・結果のドキュメント化
- 成果物: 実行前提、コマンド、成功判定、失敗時切り分けを記載した手順書。
- 確認方法: 手順書どおりに再実行し、同じ結果を再現できることを確認する。

10. 品質ゲート実行
- 成果物: 品質ゲート実行結果（format/compile/docs/credo/test --cover）。
- 確認方法: `docs/common/品質ゲート.md` の基準をすべて満たすことを確認する。

## 5. 品質ゲート
- 共通定義: `docs/common/品質ゲート.md` を参照する。
- 追加条件:
  - 移動・削除APIの追加時は、正常系と異常系（必須不足・型不正）の双方をテストで担保する。
  - `mix test --cover` の総合カバレッジ 90%以上を維持する。
- 例外条件:
  - 実機依存の確認は手動サンプル実行で行い、通常の単体テストから分離する。

## 6. 実行手順
- 前提:
  - Resonite 側で ResoniteLink を有効化し、表示ポートを控える。
  - 接続先は `localhost` を使用する。
  - `MIX_HOME` を本PJ用に設定する。
- コマンド:
  - `cd ~/local/vr_elixir/resonite_link_ex`
  - `export MIX_HOME=~/local/vr_elixir/.mix`
  - `mix format --check-formatted`
  - `mix compile --warnings-as-errors`
  - `mix check.docs`
  - `mix credo --strict`
  - `mix test --cover`
  - `mix run examples/<sprint3_sample>.exs -- --port <ResoniteLink表示ポート>`

## 7. リスク・詰まりポイント
- 事象: `name` 解決時に対象が見つからない、または同名が複数存在して対象が曖昧になる。
- 検知方法: 実行時に `not_found` / `ambiguous_name` が返る。
- 回避策: `name` の一意運用ルールを定義し、必要に応じて `parent_name` で探索範囲を絞る。
- 事象: `position` 形式不正（map構造・数値型不一致）で `updateSlot` が失敗する。
- 検知方法: バリデーションエラーまたは実行時失敗応答を確認する。
- 回避策: API入口で `x/y/z` 数値チェックを必須化し、異常系テストを先に追加する。
- 事象: 実機接続条件不備（ホスト/ポート）で検証が進まない。
- 検知方法: 接続失敗、404、タイムアウトが発生する。
- 回避策: `localhost` 固定・ポート再確認・最小サンプルで疎通確認後に本検証へ進む。
