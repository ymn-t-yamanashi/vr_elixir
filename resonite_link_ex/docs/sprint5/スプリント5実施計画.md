# スプリント5実施計画

## 1. 目的
- Core API の実利用導線（examples）を整備し、将来の統合テスト導入に向けた対象層と実施方針を確定する。

## 2. 対象範囲
### In
- `ResoniteLinkEx.Core` の主要コマンドを使ったサンプルプログラムを追加する。
- 既存サンプルとの役割分担（Facade API と Core API）を README / docs に明記する。
- 統合テスト導入方針を文書化し、対象層（Core / Facade）と初期対象シナリオを確定する。
- 統合テストを見据えた最小限の補助 API（必要時のみ）を実装する。
- 追加・変更分の単体テストを整備し、既存回帰がないことを確認する。

### Out
- ResoniteLink プロトコルの新規 `$type` 追加。
- 実機統合テストの本実装・CI 常時実行化。
- 既存公開 API の破壊的変更。
- GUI ツールや外部監視基盤の導入。

## 3. 完了条件（DoD）
- Core API を使ったサンプルが再実行可能な手順付きで提供される。
- 利用者が「Facade API を使う場面」と「Core API を使う場面」を README / docs で判別できる。
- 統合テスト方針ドキュメントに以下が明記されている。
  - 対象層（Core / Facade）
  - 初期対象コマンド・シナリオ
  - 前提環境と実行条件
  - 非対象（今回やらないこと）
- `mix format --check-formatted` / `mix compile --warnings-as-errors` / `mix check.docs` / `mix credo --strict` / `mix test --cover` がすべて成功する。
- 総合カバレッジが前スプリント基準（100%）を下回らない。

## 4. 実装タスク（最小単位）
1. Core サンプル要件の確定
- 成果物: 対象コマンド、前提、期待結果をまとめたメモ（`docs/sprint5` 配下）。
- 確認方法: 既存サンプルとの差分レビューで重複がないことを確認する。

2. Core API サンプル追加
- 成果物: `examples/sprint5_core_commands_sample.exs`（仮称）。
- 確認方法: `mix run examples/sprint5_core_commands_sample.exs -- --port <port>` で手順どおり実行できる。

3. ドキュメント導線更新
- 成果物: `README.md` または `docs/README.md` の利用ガイド追記（Facade/Core の使い分け）。
- 確認方法: 新規利用者が最短でサンプル実行手順に到達できることをレビューで確認する。

4. 統合テスト方針の確定
- 成果物: `docs/sprint5/統合テスト方針.md`（対象層・初期シナリオ・非対象を明文化）。
- 確認方法: スプリント6着手時にそのままタスク分解できる粒度になっていることを確認する。

5. テスト整備と品質ゲート
- 成果物: 必要な単体テスト追加とゲート実行ログ。
- 確認方法: 品質ゲート一式の成功を確認する。

## 5. 品質ゲート
- 共通定義: `docs/common/品質ゲート.md` を参照する。
- 追加条件:
  - サンプル追加時は「失敗時の復旧方法（接続先未起動時の案内）」をコメントまたは docs に必ず記載する。
  - 方針ドキュメントは「対象」と「非対象」を同時に記載し、スコープ膨張を防止する。

## 6. 実行手順
- 前提:
  - `cd ~/local/vr_elixir/resonite_link_ex`
  - `export MIX_HOME=~/local/vr_elixir/.mix`
  - ResoniteLink 側で受信ポートを確認済みであること。
- コマンド:
  - `cd ~/local/vr_elixir/resonite_link_ex`
  - `export MIX_HOME=~/local/vr_elixir/.mix`
  - `mix deps.get`
  - `mix format`
  - `mix compile --warnings-as-errors`
  - `mix check.docs`
  - `mix credo --strict`
  - `mix test --cover`
  - `mix run examples/sprint5_core_commands_sample.exs -- --port <ResoniteLink表示ポート>`

## 7. リスク・詰まりポイント
- 事象: Core サンプルが低レベルすぎて利用者の理解コストが高くなる。
- 検知方法: サンプル読了後に必要前提（sessionData、slot_id など）の質問が多発する。
- 回避策: サンプル冒頭で「前提データ取得 → コマンド実行」の流れを固定し、コメントを最小限付与する。
- 事象: 統合テスト方針の粒度不足で次スプリントの着手が止まる。
- 検知方法: 方針ドキュメントからタスク分解時に追加ヒアリングが必要になる。
- 回避策: 各シナリオに「入力・期待結果・失敗時判定」を最低1行ずつ記載する。

## 8. 終了時チェック
- 終了判定:
  - DoD（Core サンプル、導線、統合テスト方針、品質ゲート、カバレッジ維持）をすべて満たしている。
  - 追加サンプルと既存サンプルの役割重複がない。
- 次に決めること:
  - 実機統合テストをローカル専用で先行導入するか、CI 連携前提で段階導入するか。
  - 統合テストの最初の自動化対象を `Core` 優先にするか `Facade` 優先にするか。
