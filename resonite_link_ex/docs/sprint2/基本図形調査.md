# 基本図形調査（スプリント2）

## 目的
- ResoniteLink 公式情報を一次情報として、基本図形生成で必要な仕様を確認する。

## 確認日
- 2026-05-05

## 公式確認結果（一次情報）

### 1. 図形生成は `addComponent` の `componentType` 指定で行う
- 公式ドキュメントに `addComponent` の仕様がある。
- `data.componentType` は必須で、Resonite のコンポーネントアタッチャと同じ型名構文を使う。
- 根拠:
  - `docs/docs/websocket/components.md`
  - `README.md` の `addComponent` 説明

### 2. ResoniteLink 側は図形名の固定一覧を持たない
- 公式メッセージ定義は `addComponent` / `updateComponent` などの命令定義のみで、図形コンポーネント名の固定列挙はない。
- 根拠:
  - `ResoniteLink/Models/Messages/Message.cs`

### 3. 利用可能コンポーネント一覧は `getComponentTypeList` で取得する設計
- `getComponentTypeList` が定義され、`categoryPath: "*"` で全コンポーネント一覧を取得できる。
- つまり「どの基本図形が使えるか」は実行環境の一覧取得で確定するのが公式想定。
- 根拠:
  - `ResoniteLink/Models/Messages/Reflection/GetComponentDefinitionList.cs`
  - `ResoniteLink/LinkInterface.cs`（`GetAllComponentTypes()` が `ALL_COMPONENTS` を使用）

## 現時点の判断
- 公式情報だけでは「基本図形名の確定リスト」は取得できない。
- スプリント2では次の手順で確定する。
  1. `getComponentTypeList` でコンポーネント一覧を取得する。
  2. 図形候補（quad/cube/sphere/cylinder/capsule/ring/grid）に対応する `componentType` を抽出する。
  3. `addComponent` で実機生成を確認し、採用可否を判定する。

## 次アクション
- 実機で `getComponentTypeList` を呼び、基本図形候補の正式 `componentType` を確定する。

## 実機試行結果（2026-05-05）
- 実行コマンド:
  - `mise exec -- mix run examples/list_component_types.exs -- --port 44889`
- 結果:
  - `getComponentTypeList` 呼び出し自体は `success: true`
  - 生レスポンス:
    - `$type: "componentTypeList"`
    - `componentTypes: []`
    - `subcategories: []`
    - `totalComponentCount: 0`
  - 一覧取得は成功したが、返却内容は空配列だった。

## 現時点の判断
- 取得処理の問題ではなく、Resonite 側が `componentTypeList` を空で返している状態と判断できる。
- `categoryPath: "*"` でも空のため、セッション状態・権限・対象ワールド条件の影響を確認する必要がある。

## 次ステップ（確認項目）
1. Resonite 側で対象セッションが編集可能状態か確認する。
2. ResoniteLink 有効化直後に再取得して差分を確認する。
3. `getComponentTypeList` の `categoryPath` を空文字または既知カテゴリで試し、応答差を確認する。
