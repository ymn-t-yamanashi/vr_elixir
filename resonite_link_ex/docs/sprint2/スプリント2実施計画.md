# スプリント2実施計画

## 1. 目的
- このPJのElixirライブラリ（ResoniteLinkEx）を使い、基本図形（quad・cube・sphere・cylinder・capsule・ring・grid）を実機で再現可能な手順で生成できる状態にする。

## 2. 対象範囲
### In
- スプリント1で確立した代表命令基盤（接続/送信/応答処理）を使い、`quad` `cube` `sphere` `cylinder` `capsule` `ring` `grid` を生成できるようにする。
- 各図形の生成に必要な最小パラメータ（位置・スケール・色）を定義し、再現可能な実行手順を整備する。
- サンプル実行で 7 図形の生成成功を確認できるようにする。
- ResoniteLink経由で生成可能な基本図形候補を調査し、スプリント2時点の対象一覧を確定する。
- 基本図形を生成する専用モジュールの設計（責務・I/F・エラー方針）を作成する。

### Out
- 上記7図形以外の図形対応。
- 高度な材質設定（テクスチャ読み込み、複雑なPBR設定）。
- 全命令対応や汎用アセット取り込み最適化。

## 3. 完了条件（DoD）
- `mix format --check-formatted` / `mix compile --warnings-as-errors` / `mix check.docs` / `mix credo --strict` / `mix test --cover` がすべて成功する。
- `mix test --cover` の総合カバレッジが 90%以上を満たし、100%を目標として維持する。
- `quad` `cube` `sphere` `cylinder` `capsule` `ring` `grid` の7図形が、実機でサンプル実行により生成成功する。
- 各図形の生成手順（前提・実行コマンド・確認観点）がドキュメント化され、同手順で再現できる。
- `docs/sprint2/基本図形調査.md` に、調査対象・結果・採用図形・見送り理由が記録されている。
- `docs/sprint2/基本図形モジュール設計.md` に、モジュール構成・公開関数・引数仕様・戻り値仕様・エラー方針が記録されている。

## 4. 実装タスク（最小単位）
1. 基本図形調査
- 成果物: `docs/sprint2/基本図形調査.md`（候補一覧、検証結果、採用/見送り判定）。
- 確認方法: 記録した検証コマンドと結果ログが対応していることを確認する。

2. 基本図形の採用範囲確定
- 成果物: スプリント2で実装対象とする図形一覧（`quad` `cube` `sphere` `cylinder` `capsule` `ring` `grid`）と確定理由。
- 確認方法: `対象範囲 In/Out` と `基本図形調査.md` の判定が一致していることを確認する。

3. 基本図形モジュール設計
- 成果物: `docs/sprint2/基本図形モジュール設計.md`（責務、I/F、戻り値、エラー方針）。
- 確認方法: 設計内容がスプリント2のIn/DoDと矛盾しないことをレビューで確認する。

4. 図形生成APIの追加
- 成果物: `quad` `cube` `sphere` `cylinder` `capsule` `ring` `grid` を生成する関数、または既存サンプル内で7図形を生成する共通化処理。
- 確認方法: 単体テストで生成payloadの必須項目を確認し、送信データが期待形式であることを確認する。

5. サンプル実行フローの整備
- 成果物: 7図形を順に生成できる実行サンプル（ポート引数指定必須、`localhost` 固定）。
- 確認方法: `mix run` 実行で各命令の成功応答を確認する。

6. 実機検証手順のドキュメント化
- 成果物: 前提条件、実行コマンド、成功判定、失敗時切り分けを含む手順書。
- 確認方法: 手順書どおりに再実行し、同じ結果を再現できることを確認する。

7. 品質ゲート実行
- 成果物: 品質ゲート実行ログ（format/compile/docs/credo/test --cover の成功結果）。
- 確認方法: 共通基準 `docs/common/品質ゲート.md` の全項目が成功していることを確認する。

8. 終了時チェック
- 成果物: DoD確認結果と次スプリント検討項目の記録。
- 確認方法: `完了条件（DoD）` の各項目に対して達成有無を明記する。

## 5. 品質ゲート
- 共通定義: `docs/common/品質ゲート.md` を参照する。
- 追加条件:
  - 図形生成サンプル変更時は `mix test --cover` 後に実機サンプル実行ログで成功応答を確認する。
  - カバレッジは総合90%以上を必須、100%維持を目標とする。
- 例外条件:
  - 実機依存の確認は `@tag integration: true` または手動実行手順で実施し、通常 `mix test` に混在させない。

## 6. 実行手順
- 前提:
  - Resonite 側で ResoniteLink を有効化し、表示されたポートを確認する。
  - 接続先ホストは `localhost` 固定（`127.0.0.1` は使用しない）。
  - `MIX_HOME` を本PJ用に指定する。
- コマンド:
  - `cd ~/local/vr_elixir/resonite_link_ex`
  - `export MIX_HOME=~/local/vr_elixir/.mix`
  - `mix local.hex --force`
  - `mix local.rebar --force`
  - `mix deps.get`
  - `mix format --check-formatted`
  - `mix compile --warnings-as-errors`
  - `mix check.docs`
  - `mix credo --strict`
  - `mix test --cover`
  - `mix run examples/red_square_sample.exs -- --port <ResoniteLink表示ポート>`

## 7. リスク・詰まりポイント
- 事象: `localhost` 以外のホスト指定で `404` または接続失敗になる。
- 検知方法: 実行時に `WebSockex.RequestError`（404）または `ConnError` が出る。
- 回避策: 接続先を `localhost` に固定し、ポートは Resonite UI 表示値を都度使用する。
- 事象: Hex/Mix 環境不整合で `Hex.Repo` 起動失敗が発生する。
- 検知方法: `please re-compile this module with an Erlang/OTP 28 compiler` 等のエラーが出る。
- 回避策: `MIX_HOME=~/local/vr_elixir/.mix` を設定し、`mix local.hex --force` を再実行する。
- 事象: 実機応答形式差（`sourceMessageId`）によるデコード失敗。
- 検知方法: `decode_error` ログが連続して出る。
- 回避策: 実応答フォーマットを優先し、受信正規化のテストを維持する。

## 8. 終了時チェック
- 終了判定:
  - DoDの全項目（品質ゲート、7図形生成、手順再現）を満たしていることを確認する。
  - 実行ログと更新ドキュメントが `docs/sprint2` 配下に揃っていることを確認する。
- 次に決めること:
  - スプリント3で対象とする図形/命令の拡張範囲。
  - 実機統合テストの自動化範囲（どこまで `integration` タグ化するか）。
