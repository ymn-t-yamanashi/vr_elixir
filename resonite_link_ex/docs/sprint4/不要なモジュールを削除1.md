# 不要なモジュール削除 調査1

## 調査対象
- `ResoniteLinkEx.Scene` モジュール
- `ResoniteLinkEx.call/3` 関数

## 結論（先に要点）
- `ResoniteLinkEx.Scene` は **現時点では削除不可**。
- `ResoniteLinkEx.call/3` は **現時点では削除非推奨（互換APIとして残すのが安全）**。

## 根拠

### 1. `ResoniteLinkEx.Scene` の利用状況
- 実装で直接参照あり
  - `lib/resonite_link_ex/core.ex` が `alias ResoniteLinkEx.Scene` し、各Core関数が `Scene.call/3` に依存。
- テストで直接参照あり
  - `test/resonite_link_ex/scene_test.exs`。
- ドキュメントにも公開APIとして記載あり
  - `README.md`
  - `docs/design/詳細設計.md`

判定:
- `Scene` を削除すると `Core` が成立しなくなるため、**先にCore内部実装の差し替えが必要**。

### 2. `ResoniteLinkEx.call/3` の利用状況
- 実装本体
  - `lib/resonite_link_ex.ex` に公開関数として定義（内部は `Client.call/3` 委譲）。
- テスト参照あり
  - `test/resonite_link_ex_test.exs`。
- READMEに公開APIとして記載あり
  - `README.md`。

判定:
- `call/3` は「任意 `$type` を透過的に呼べる入口」で、`Core` 個別関数群とは役割が異なる。
- 即削除すると利用者互換性を壊す可能性が高いため、**段階的非推奨化なしの削除は不可**。

## 削除したい場合の現実的な手順
1. `Scene` 依存を外す
- `Core` が `Scene.call/3` に依存しない構成へ変更（例: `Protocol` 直接利用など）。
- `scene_test.exs` を削除または置換。

2. `ResoniteLinkEx.call/3` を段階的に整理
- `@doc` で非推奨化を明示。
- READMEの主要導線を `Core` / 高レベルAPI（Objects/Shapes）へ移す。
- 利用箇所の移行完了後に削除。

3. ドキュメント整合
- `README.md` と `docs/design/詳細設計.md` から `Scene` / `call/3` 記述を更新。

## 最終判定
- 今回の調査時点では、両対象とも「不要」と断定できず、**削除は時期尚早**。
- 先に「依存除去」と「互換移行」を完了させるのが妥当。

## リスクと対策
- リスク:
  - `call/3` 削除で外部利用者が破壊的変更を受ける。
- 対策:
  - 1リリースは `@deprecated` 運用し、削除は次メジャーへ送る。

- リスク:
  - `Scene` のバリデーション仕様を `Core` へ移した際に挙動差分が出る。
- 対策:
  - 既存テストに加え、異常系（型不正・payload不正）ケースを追加して同値性を担保。
