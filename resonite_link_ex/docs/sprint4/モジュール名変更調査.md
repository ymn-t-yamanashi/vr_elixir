# モジュール名変更調査（`ResoniteLinkEx` -> `ResoniteLinkEx.Slot`）

## 1. 調査目的
- ルート公開モジュール `ResoniteLinkEx` を `ResoniteLinkEx.Slot` に変更できるか確認する。
- 変更時の影響範囲と移行手順を整理する。

## 2. 結論
- **そのままの一括リネームは非推奨**。
- 理由: 現在の `ResoniteLinkEx` は Slot 操作専用ではなく、以下を含む総合エントリポイントのため。
  - クライアント起動 (`start_client/1`)
  - レスポンス処理 (`receive_response/2`)
  - オブジェクト操作委譲 (`move_slot_by_name/4`, `delete_slot_by_name/3`, 互換API)
  - 図形生成 (`spawn_shape/3`)
  - ポート検出 (`find_resonite_link_port/0,1`)

## 3. 実コード影響範囲

### 3.1 直接影響（公開API本体）
- `lib/resonite_link_ex.ex`
  - `defmodule ResoniteLinkEx do` を変更する場合、公開API名がすべて破壊的変更になる。

### 3.2 テスト影響
- `test/resonite_link_ex_test.exs`
  - `doctest ResoniteLinkEx`
  - `ResoniteLinkEx.*` 呼び出しが多数あり、全面修正が必要。
- ほか複数テストで `ResoniteLinkEx.start_client/0` 等を参照。

### 3.3 ドキュメント影響
- `README.md`
  - 公開API一覧が `ResoniteLinkEx.*` 前提。
- `docs/design/詳細設計.md`
  - モジュール分割・公開APIの記載更新が必要。
- スプリント設計/調査ドキュメントにも `ResoniteLinkEx` 記述が多数存在。

### 3.4 間接影響
- 利用者コード（外部プロジェクト）で `ResoniteLinkEx` を参照している場合、全破壊的変更。

## 4. 技術的論点
- `ResoniteLinkEx.Slot` という名前は「Slot専用モジュール」を示す。
- しかし現 `ResoniteLinkEx` は Slot 以外の責務（Shapes/PortDiscovery/Client facade）を持つ。
- したがって、**名前と責務が不一致** になる。

## 5. 実施するなら推奨方針

### 方針A（推奨）: ルートは維持し、Slot専用を追加
1. `ResoniteLinkEx.Slot` を新設し、Slot操作API（例: move/delete 系）を集約。
2. 既存 `ResoniteLinkEx` は後方互換Facadeとして残す。
3. `ResoniteLinkEx` から `ResoniteLinkEx.Slot` へ委譲。
4. `@deprecated` を段階的に付与し、次メジャーで整理。

メリット:
- 破壊的変更を抑えられる。
- 名前と責務を段階的に一致させられる。

### 方針B（非推奨）: ルートを直接 `ResoniteLinkEx.Slot` に改名
- すべての呼び出し、doctest、README、設計書、外部利用者に一斉変更が必要。
- 互換性コストが大きく、回帰リスクが高い。

## 6. 最小リスク移行計画（案）
1. `ResoniteLinkEx.Slot` 新設（既存 Objects API の再公開）。
2. `ResoniteLinkEx` 側に移行ガイド付き `@doc` 追加。
3. README の主要導線を `ResoniteLinkEx.Slot` 優先に変更。
4. 内部テストを `ResoniteLinkEx.Slot` 呼び出し中心へ切替。
5. 1リリース運用後、不要Facadeを次メジャーで削除判断。

## 7. 判定
- 現時点では、`ResoniteLinkEx` を直接 `ResoniteLinkEx.Slot` に置換するのではなく、
  **`ResoniteLinkEx.Slot` の追加 + 段階移行** が妥当。
