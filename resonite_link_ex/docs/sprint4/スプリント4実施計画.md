# スプリント4実施計画

## 1. 目的
- `mix docs` で利用者が API を追えるドキュメントを整備し、ResoniteLink の基礎コマンドを直接呼び出せる Core API を提供する。

## 2. 対象範囲
### In
- 公開 API/主要モジュールの `@moduledoc` / `@doc` を利用者視点に改修する（引数、戻り値、使用例、失敗条件）。
- `mix docs` 生成に必要な `mix.exs` / `README` / ドキュメント導線を整備する。
- 基礎コマンドを直接呼び出せる Core モジュール（例: `ResoniteLinkEx.Core`）を追加する。
- Core モジュールで下記コマンドの薄い委譲 API を提供する。
  - `requestSessionData`
  - `addSlot`
  - `updateSlot`
  - `addComponent`
  - `updateComponent`
  - `removeComponent`
  - `removeSlot`
  - `getSlot`
- Core API の正常系・異常系テストを追加し、既存 API との整合性を確認する。
- 既存 API（`ResoniteLinkEx`）との責務境界を明文化する（README か docs）。

### Out
- ResoniteLink プロトコル自体の拡張（新規 `$type` 追加）。
- 実機連携の統合テスト自動化。
- 図形生成ロジックや Name 解決ロジックの仕様変更。
- 既存 API の破壊的変更・削除。

## 3. 完了条件（DoD）
- `mix docs` が成功し、主要公開 API に説明不足がない（用途、引数、返り値、エラー条件）。
- Core モジュール経由で8コマンドを直接呼び出せることを単体テストで確認できる。
- Core API の不正入力時に `{:error, :invalid_request}` など既存ポリシーで返る。
- 既存テストが回帰しない（`mix test` 0 failures）。
- 品質ゲート（format/compile/check.docs/credo/test --cover）を通過する。
- カバレッジ目標を維持する（現状 100% を下回らない）。

## 4. 実装タスク（最小単位）
1. ドキュメント方針確定と対象 API の棚卸し
- 成果物: 改修対象一覧（モジュール/関数）と記述方針（何を書くか）。
- 確認方法: `lib/` 公開関数と既存 `@doc` の差分レビュー。

2. `mix docs` 導入・生成設定
- 成果物: `mix.exs` の docs 設定、必要なら docs 用依存追加、README 導線。
- 確認方法: `mix docs` が成功し HTML が生成される。

3. 公開ドキュメント改修
- 成果物: `ResoniteLinkEx` / `Protocol` / `Client` / `Objects` など主要 API の `@doc` 改修。
- 確認方法: `mix docs` 生成物で API の使い方が追えることを確認する。

4. Core モジュール新設
- 成果物: `ResoniteLinkEx.Core`（仮）と 8 コマンド直接呼び出し関数。
- 確認方法: 各関数が既存の送信基盤へ委譲されることをテストで確認する。

5. Core API テスト追加
- 成果物: 正常系/異常系テスト（型不正、必須不足、未知コマンドなど）。
- 確認方法: `mix test` で追加ケースが通り、既存テストも回帰しない。

6. 品質ゲート実行と最終調整
- 成果物: 品質ゲート通過ログ、必要最小限の修正コミット。
- 確認方法: `mix format --check-formatted` / `mix compile --warnings-as-errors` / `mix check.docs` / `mix credo --strict` / `mix test --cover` がすべて成功。

## 5. 品質ゲート
- 共通定義: `docs/common/品質ゲート.md` を参照する。
- 追加条件:
  - `mix docs` 成功をスプリント4の必須ゲートに追加する。
  - Core API 追加分は正常系と異常系を必ず同時にテスト追加する。
  - 総合カバレッジ 100% を維持する。

### mix docs ルール（サンプル付き）
1. `@moduledoc` は「何をするモジュールか」を先頭1〜2文で明記する。
2. 公開関数は原則すべて `@doc` を付与し、次の順で記載する。
  - 目的（何をするか）
  - 引数（必須/任意、期待する型）
  - 戻り値（成功時/失敗時）
  - 代表的な失敗条件（`invalid_request` など）
  - `iex>` で始まる実行例（可能な範囲）
3. 実装詳細（内部状態や private 関数都合）は `@doc` に書かず、利用者に必要な振る舞いに限定する。
4. `@spec` と `@doc` の整合を必須にする（型・戻り値が食い違わないこと）。
5. docs 生成物で関数一覧を見た時に、関数名だけでなく用途差分が判別できる説明にする。
6. 非推奨 API は `@doc` の先頭に「互換API/将来削除予定」を明記する。

#### 良い例
```elixir
@doc """
`name` で対象 Slot を解決し、座標を更新します。

## Parameters
- `client_or_transport`: `pid()`。Client または Transport の PID。
- `name`: `String.t()`。更新対象の Slot 名。
- `position`: `%{"x" => number(), "y" => number(), "z" => number()}`。

## Returns
- `{:ok, map()}`: `updateSlot` リクエスト生成/送信に成功。
- `{:error, :invalid_request | :not_found | :ambiguous_name | term()}`: 入力不正または解決/送信失敗。

## Examples
    iex> position = %{"x" => 0, "y" => 1, "z" => 2}
    iex> ResoniteLinkEx.move_slot_by_name(client, "CubeA", position)
    {:ok, _request}
"""
@spec move_slot_by_name(term(), String.t(), map(), keyword()) ::
        {:ok, map()} | {:error, term()}
```

#### 悪い例
```elixir
@doc "移動する"
def move_slot_by_name(client, name, position, opts), do: ...
```
- 何をどう移動するか不明。
- 引数型・戻り値・失敗条件が不明。
- 利用例がなく、呼び出し方を判断できない。

### ルール根拠（文法エビデンス）
以下は「Elixir/ExDoc の公式仕様」に基づく必須事項。

1. `@moduledoc` / `@doc` を使ってモジュール・関数ドキュメントを記述する。
- 根拠: Elixir公式 Module attributes
- https://hexdocs.pm/elixir/module-attributes.html

2. `@doc` / `@moduledoc` 本文は Markdown として記述する。
- 根拠: Elixir公式 Writing documentation
- https://hexdocs.pm/elixir/writing-documentation.html

3. `iex>` / `...>` 形式のサンプルは doctest 構文として扱える。
- 根拠: ExUnit.DocTest 公式
- https://hexdocs.pm/ex_unit/ExUnit.DocTest.html

4. `mix docs` は ExDoc によりドキュメントを生成する。
- 根拠: ExDoc `mix docs` タスク
- https://hexdocs.pm/ex_doc/Mix.Tasks.Docs.html

### PJ運用ルール（公式文法ではなく、本PJの品質基準）
以下は公式仕様を補完するための本PJ独自ルール。

1. `@doc` は「目的 → 引数 → 戻り値 → 失敗条件 → 例」の順で書く。
2. `@spec` と `@doc` の型・戻り値表現を一致させる。
3. 非推奨 API は `@doc` 冒頭で「互換API/将来削除予定」を明記する。
4. Core API 追加時は、同時に docs とテスト（正常系/異常系）を追加する。
5. 実装者は、実装完了時に「ルール本文」と「実装中の `@doc`」を突合し、不一致が1件でもあれば修正してから完了報告する。
6. `## Examples` は下記の統一フォーマットを必須とする。
  - `iex> <前提値の準備>`
  - `iex> <API呼び出し>`
  - `<期待結果>`
7. ルール不一致を指摘された場合は、言い訳せずに次の順で対応する。
  - 不一致箇所を列挙する。
  - ルール準拠へ修正する。
  - 再発防止のチェック項目をルールへ追記する。

補足（`## Parameters` / `## Returns` について）:
- `## Parameters` / `## Returns` という見出し名は、Elixir公式の必須キーワードではない。
- 公式仕様上の要件は「Markdown を使うこと」「新しい節は `##` 見出しで始めること」。
- したがって本PJでは、可読性統一のために見出し名を `## Parameters` / `## Returns` に統一する運用とする。
- 根拠:
  - https://hexdocs.pm/elixir/writing-documentation.html
  - https://hexdocs.pm/ex_doc/ExDoc.Markdown.html

## 7. リスク・詰まりポイント
- 事象: `@doc` の粒度が揃わず、利用者が API を辿れない。
- 検知方法: docs 生成後の目視レビューで前提や戻り値の説明欠落が見つかる。
- 回避策: 先に記述テンプレート（目的/引数/戻り値/例/失敗条件）を固定する。
- 事象: Core API と既存 `ResoniteLinkEx` API の責務重複で命名が曖昧になる。
- 検知方法: 既存 API と新 API の使い分け説明が1文で言えない。
- 回避策: Core は「生コマンド委譲」、既存 API は「ユースケース API」と責務を分離する。
- 事象: カバレッジ 100% が低下する。
- 検知方法: `mix test --cover` の総合値が 100% を下回る。
- 回避策: 追加関数ごとに同時にテストを足し、未到達分岐をその場で解消する。
